# Rules: ISO 25010 — Software Quality Model

> Source: ISO/IEC 25010:2023
> Version: 2023
> Applicability: All software systems

## Overview

ISO 25010 defines 8 quality characteristics for software products. Each characteristic has
sub-characteristics that provide specific quality attributes to evaluate.

## The 8 Quality Characteristics

### 1. Functional Suitability

> Degree to which a product or system provides functions that meet stated and implied needs
> when used under specified conditions.

| Sub-characteristic | Definition | How to Evaluate |
|-------------------|------------|-----------------|
| **Functional completeness** | Degree to which the set of functions covers all specified tasks | Check: Does the system handle all business requirements? Are there missing features? |
| **Functional correctness** | Degree to which a product provides correct results | Check: Do calculations produce expected outputs? Are edge cases handled? |
| **Functional appropriateness** | Degree to facilitate accomplishment of specified tasks | Check: Is the workflow intuitive? Are there unnecessary steps? |

**Code-Level Checks:**
- Business logic produces correct results for known inputs
- Edge cases (empty collections, null values, zero divisions) handled
- All specified user stories are implementable with existing code
- Missing functionality identified (stub methods, TODO comments, empty directories)

### 2. Performance Efficiency

> Performance relative to the amount of resources used under stated conditions.

| Sub-characteristic | Definition | How to Evaluate |
|-------------------|------------|-----------------|
| **Time behaviour** | Response times, processing times, throughput | Check: N+1 queries, unbounded queries, missing indexes |
| **Resource utilization** | CPU, memory, disk, network usage | Check: Memory leaks, large collections in memory, file handle leaks |
| **Capacity** | Maximum limits of a product parameter | Check: Pagination, limits on concurrent users, upload sizes |
| **Efficiency trade-offs** | Balance between performance and other characteristics | Check: Is the code optimized for the right things? |

**Code-Level Checks:**
```php
// N+1 query detection
$users = User::all();
foreach ($users as $user) {
    $user->posts; // N+1! Should use with('posts')
}

// Missing pagination
$allRecords = Model::all(); // Should be ->paginate()

// Unbounded queries
$largeCollection = Model::where('active', true)->get(); // May return millions
```

### 3. Compatibility

> Degree to which a product can exchange information with other products and/or perform
> its required functions while sharing the same hardware or software environment.

| Sub-characteristic | Definition | How to Evaluate |
|-------------------|------------|-----------------|
| **Co-existence** | Can operate alongside other products without interference | Check: Does the app conflict with other services on the same server? |
| **Interoperability** | Can exchange information with other products | Check: API compatibility, data format standards, import/export |

**Code-Level Checks:**
- PHP version compatibility (no deprecated functions used)
- Database compatibility (no vendor-specific SQL)
- Browser compatibility (no browser-specific CSS/JS without fallbacks)
- API responses follow standard formats (JSON:API, HAL, etc.)

### 4. Usability

> Degree to which a product can be used by specified users to achieve specified goals with
> effectiveness, efficiency, and satisfaction.

| Sub-characteristic | Definition | How to Evaluate |
|-------------------|------------|-----------------|
| **Appropriateness recognisability** | Users can recognize if the product is right for them | Check: Clear purpose, obvious functionality |
| **Learnability** | Product can be used without assistance | Check: Intuitive navigation, consistent patterns |
| **Operability** | Product has attributes that make it easy to operate | Check: Forms have labels, errors are helpful, undo exists |
| **User error protection** | System protects users from errors | Check: Confirmation dialogs for destructive actions, input validation |
| **User engagement** | Product presents functions in an inviting way | Check: Consistent UI, no jarring transitions |
| **Inclusivity** | Can be used by people with widest range of characteristics | Check: Accessibility, keyboard navigation, screen readers |
| **User assistance** | Can be used with user assistance features | Check: Help text, tooltips, documentation links |

**Code-Level Checks:**
- All form inputs have labels
- Error messages are human-readable (not technical)
- Destructive actions require confirmation
- Loading states shown during async operations
- Empty states handled (what to do when no data exists)

### 5. Reliability

> Degree to which a system performs specified functions under specified conditions for a
> specified period of time.

| Sub-characteristic | Definition | How to Evaluate |
|-------------------|------------|-----------------|
| **Maturity** | Meets needs under normal operation | Check: Error handling, exception management, graceful degradation |
| **Availability** | System is operational and accessible | Check: Uptime, maintenance mode, health checks |
| **Fault tolerance** | Operates despite faults | Check: Try/catch blocks, fallback mechanisms, retry logic |
| **Recoverability** | Can recover data and state after failure | Check: Database transactions, backups, session recovery |

**Code-Level Checks:**
```php
// BAD: No error handling
$user = User::find($id); // What if $id is invalid?
$user->doSomething();

// GOOD: Proper error handling
try {
    $user = User::findOrFail($id);
    $user->doSomething();
} catch (ModelNotFoundException $e) {
    Log::warning('User not found', ['id' => $id]);
    return redirect()->route('users.index')
        ->with('error', __('User not found.'));
}
```

```php
// BAD: No transaction for multi-step mutation
$order->save();
$inventory->save();
$payment->save();

// GOOD: Transaction wraps all mutations
DB::transaction(function () use ($order, $inventory, $payment) {
    $order->save();
    $inventory->save();
    $payment->save();
});
```

### 6. Security

> Degree to which a product protects information and data.

| Sub-characteristic | Definition | How to Evaluate |
|-------------------|------------|-----------------|
| **Confidentiality** | Data is accessible only to authorized users | Check: Authorization, encryption, access control |
| **Integrity** | Prevents unauthorized access/modification | Check: Input validation, CSRF, checksums |
| **Non-repudiation** | Actions can be proven to have taken place | Check: Audit logging, activity trail |
| **Accountability** | Actions traceable to a user | Check: User identification in logs, audit trail |
| **Authenticity** | Identity can be proven | Check: Authentication, MFA, token validation |

**Note:** Security details are in `owasp-top10.md` and `authentication-authorization.md`.
This section covers the ISO 25010 framing only.

### 7. Maintainability

> Degree of effectiveness and efficiency with which a product can be modified.

| Sub-characteristic | Definition | How to Evaluate |
|-------------------|------------|-----------------|
| **Modularity** | System composed of discrete components | Check: Module boundaries, single responsibility, loose coupling |
| **Reusability** | Assets can be used in multiple systems | Check: Base classes, shared services, traits |
| **Analysability** | Can be assessed for impact of changes | Check: Clear interfaces, typed properties, documentation |
| **Modifiability** | Can be modified without introducing defects | Check: SOLID principles, dependency injection, test coverage |
| **Testability** | Can be effectively tested | Check: Test structure, mocking support, isolation |

**Code-Level Checks:**
- No God classes (>300 lines)
- No circular dependencies between modules
- Methods have single responsibility (<20 lines ideal)
- Classes have single responsibility (<300 lines ideal)
- No more than 3 levels of nesting
- No more than 5 parameters per method (prefer DTOs)
- No magic numbers or strings (use constants/enums)

### 8. Portability

> Degree of effectiveness and efficiency with which a system can be transferred from one
> environment to another.

| Sub-characteristic | Definition | How to Evaluate |
|-------------------|------------|-----------------|
| **Adaptability** | Can be adapted for different environments | Check: Environment variables, configuration externalization |
| **Installability** | Can be installed/uninstalled | Check: Setup scripts, migration system, documentation |
| **Replaceability** | Can replace another product for same purpose | Check: Standard interfaces, minimal vendor lock-in |

**Code-Level Checks:**
- Configuration in `.env` / config files (not hardcoded)
- No platform-specific paths (use ` storage_path()`, `base_path()`)
- No OS-specific commands
- Database-agnostic queries (no raw MySQL-specific SQL)
- Documented setup/installation process

## Evaluation Matrix

For each characteristic, assign a score from 1-5:

| Score | Rating | Definition |
|-------|--------|------------|
| 5 | Excellent | Fully meets all sub-characteristics |
| 4 | Good | Mostly meets; minor gaps |
| 3 | Adequate | Meets core requirements; notable gaps |
| 2 | Poor | Significant deficiencies |
| 1 | Very Poor | Fails to meet requirements |

## Priority Weights

Not all characteristics are equally important for every project. Suggested weights for
a web application:

| Characteristic | Weight | Rationale |
|---------------|--------|-----------|
| Security | 25% | Data protection is paramount |
| Reliability | 20% | Users depend on the system daily |
| Functional Suitability | 20% | Must do what it promises |
| Maintainability | 15% | Long-term viability |
| Usability | 10% | User adoption depends on it |
| Performance | 5% | Important but secondary to correctness |
| Compatibility | 3% | Usually acceptable with modern stack |
| Portability | 2% | Self-hosted; environment is controlled |
